\begin{frame}[fragile]
You can add two Series together, which returns a union of the two Series with the addition occurring on the shared index values. Values on either Series that did not have a shared index will produce a NULL/NaN (not a number).
\begin{lstlisting}
print(cities[['Chicago', 'New York', 'Portland']])
print('\n')
print(cities[['Austin', 'New York']])
print('\n')
print(cities[['Chicago', 'New York', 'Portland']] + cities[['Austin', 'New York']])
\end{lstlisting}
\end{frame}

\begin{frame}[fragile]\frametitle{Intuition: Index Alignment}
\begin{block}{Intuition}
Adding two Series is like matching up name tags at two different parties and adding values only
where the \emph{same} name appears at both. Anyone who only showed up to one party (an index
value present in just one Series) gets a NaN back -- there's nothing on the other side to add
them to.
\end{block}

\begin{frame}[fragile]\frametitle{Quick Check: Series}
\begin{block}{Think About It}
\lstinline{cities['Chicago']} returns a single value, but
\lstinline{cities[['Chicago', 'Portland']]} returns a Series. What's the difference in the
syntax, and why does it matter?
\end{block}
\pause
\begin{block}{Answer}
A single string key does scalar/label lookup and returns \emph{one} value. A \emph{list} of
keys is fancy indexing and always returns a Series, even with just one element -- the double
brackets are doing two things at once: the outer \lstinline{[]} is the indexing operation, the
inner \lstinline{[]} is a literal Python list being passed as the key.
\end{block}
\end{frame}

\begin{frame}[fragile]\frametitle{Intuition: Label vs. Position}
\begin{block}{Intuition}
Label-based selection is like finding a book by its title on the shelf -- you ask for
``Wednesday's row'' or ``column A,'' and pandas looks up that name. Position-based selection is
like grabbing the third book from the left, regardless of what it's called. Both answer valid
but different questions, so pandas gives you separate ways to ask each one instead of guessing
your intent.
\end{block}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Quick Check: DataFrames}
\begin{block}{Think About It}
\lstinline{df['A']} returns a Series (a column), but \lstinline{df[0:2]} returns a DataFrame
(the first two rows). Same \lstinline{[]} operator, same object -- why does pandas treat a
string key and a slice so differently?
\end{block}
\pause
\begin{block}{Answer}
The plain \lstinline{[]} operator on a DataFrame is intentionally overloaded: a single label
looks up a \emph{column} (matching the \lstinline{df.A} shortcut), while a slice looks up
\emph{rows}, mirroring plain Python/NumPy slicing. Pandas infers your intent from the
\emph{type} of the key rather than a fixed row/column rule -- which is exactly why
\lstinline{.loc}/\lstinline{.iloc} exist: they remove the ambiguity by always meaning
``by label'' or ``by position,'' regardless of what kind of key you pass.
\end{block}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{GroupBy: Split-Apply-Combine}
\texttt{groupby()} splits rows into groups by a key column. Nothing is computed until you chain on an aggregate.
\begin{block}{Intuition}
Think of \texttt{groupby()} as sorting mail into pigeonholes by recipient. Nothing gets read or
totalled yet -- that's just the sorting step. The real work happens when a clerk (an aggregate
like \texttt{.mean()} or \texttt{.sum()}) processes each pigeonhole separately and the results
are stacked back into a single tray.
\end{block}
\begin{lstlisting}
df.groupby('machine_id')
<pandas.core.groupby.generic.DataFrameGroupBy object at 0x...>
\end{lstlisting}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{GroupBy with Aggregation}
Chain \texttt{.mean()}, \texttt{.sum()}, or \texttt{.agg()} to actually run the per-group computation.
\begin{lstlisting}
df.groupby('machine_id')['temperature'].mean()
M1    75.40
M2    74.55
M3    75.50
Name: temperature, dtype: float64

df.groupby('machine_id').agg({'temperature': 'mean', 'vibration_mm_s': 'max'})
            temperature  vibration_mm_s
machine_id
M1                75.40            0.041
M2                74.55            0.034
M3                75.50            0.052
\end{lstlisting}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Quick Check: GroupBy}
\begin{block}{Think About It}
\lstinline{df.groupby('machine_id')} on its own prints only an object description, not a table
of numbers. Why doesn't pandas compute anything at that point?
\end{block}
\pause
\begin{block}{Answer}
\lstinline{groupby()} only performs the \emph{split} -- it doesn't yet know what you want to
\emph{apply} to each group. Waiting until you chain on \lstinline{.mean()}, \lstinline{.sum()},
or \lstinline{.agg()} means the same grouped object can be reused for several different
aggregations without repeating the split step, and pandas never computes a result you never
asked for.
\end{block}
\end{frame}

\begin{frame}[fragile]\frametitle{Putting It Together: Machine Health Check}
The same five-step pattern -- read, inspect, clean, transform, summarize -- shows up in almost every dataset you'll touch in this course.
\begin{lstlisting}

df = df.rename(columns={'vibration': 'vibration_mm_s'})
df = df.drop(columns=['operator_notes'])
df['high_vibration'] = df['vibration_mm_s'] > 0.025

summary = df.groupby('machine_id').agg(
    avg_temp=('temperature', 'mean'),
    flagged_readings=('high_vibration', 'sum')
)
