\begin{frame}[fragile]\frametitle{What is OpenCode?}
  \begin{itemize}
    \item Open-source coding agent built for the terminal (TUI).
	\item \textbf{IDE Integration possible}:	Works with VS Code, Cursor, and any terminal-supporting IDE
	\item \textbf{GitHub Integration}: Automate issues and PRs directly from comments
    \item \textbf{Model-Agnostic:} Supports 75+ AI models (OpenAI, Claude, Groq, Ollama, and more).
      \begin{itemize}
        \item \textit{Local:} Ollama (fully offline, zero data egress)
        \item \textit{Cloud:} OpenAI, Claude, Groq, Gemini, etc.
      \end{itemize}	
    \item \textbf{Privacy-Focused:} Runs locally or uses your own API keys, your code stays on your machine.
    \item \textbf{Built With:} Rust/Go for performance; Bubble Tea for the TUI layer.
  \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Key Features}
  \begin{itemize}
    \item \textbf{Interactive TUI:} Smooth terminal UI built with Bubble Tea; keyboard-driven workflow.
    \item \textbf{Contextual Understanding:} Reads your entire repo, understands dependencies, structure, languages.
    \item \textbf{Agentic Workflows:} Plans changes, searches files, executes shell commands, modifies code autonomously.
    \item \textbf{Plan / Build Modes/Agents:} Review proposed changes before applying,  deterministic control.
    \item \textbf{Slash Commands:} \texttt{/init}, \texttt{/help}, \texttt{/models}, \texttt{/share}, and more.
    \item \textbf{Context Mentions:} \texttt{@file:}, \texttt{@workspace} for scoped AI queries.
  \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Core Integration}
  \begin{itemize}
    \item \textbf{LSP Integration:} Language Server Protocol for real-time code intelligence (completions, diagnostics, go-to-def).
    \item \textbf{MCP (Model Context Protocol):} Plug in external tools and data sources to extend agent capabilities.
    \item \textbf{ACP Support:} Agent Communication Protocol for inter-agent coordination.
    \item \textbf{Agent Skills \& Custom Tools:} Define reusable agent behaviors and wire in custom shell/API tools.
    \item \textbf{Rules \& Permissions:} Fine-grained control over what the agent can read, write, or execute.
    \item \textbf{Multi-Provider Routing:} Failover and cost-based routing across model providers.
  \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{ Getting Started}
  \begin{itemize}
    \item \textbf{First Step:} Run \texttt{/init} to ground the agent in your project.
	\item \textbf{Configure Provider}:  \lstinline|Global config ~/.config/opencode/opencode.json Or project-specific ./opencode.json|
    \item \textbf{Config Format:} TOML-based \texttt{opencode.toml}; supports project-level and global overrides.

    \item \textbf{Bring Your Own Keys:} Set \texttt{OPENAI\_API\_KEY}, \texttt{ANTHROPIC\_API\_KEY}, or point to Ollama.
    \item \textbf{Launch:} \texttt{opencode} in any project directory. Have conda env activated before.
  \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{Plan vs Build Mode}

Switch between them using the Tab key.

Plan Mode:
  \begin{itemize}
    \item Read-only - Cannot make changes to your code
    \item Analyzes and explores your codebase
    \item Suggests implementation strategies
    \item Perfect for understanding before modifying
  \end{itemize}
  
Build Mode (Default)
  \begin{itemize}
    \item Full access - Can read, write, and modify files
    \item Executes code changes
    \item Runs commands
    \item Best for actual development work
  \end{itemize}  
  
Pro Tip: Always start complex features in Plan mode to outline your approach, then switch to Build mode for implementation.  
\end{frame}

\begin{frame}[fragile]\frametitle{AGENTS.md}
For best results, create an AGENTS.md file in your project root. This helps OpenCode understand: Your project structure, Coding conventions, Preferred patterns, Technology stack.

\begin{lstlisting}
## Project: My SaaS App

## Tech Stack
- Next.js 14 with App Router
- TypeScript
- Tailwind CSS
- PostgreSQL with Prisma

## Conventions
- Use functional components
- Prefer server components when possible
- Follow REST API naming conventions
- Write tests for all new features

## Structure
- /app - Next.js app router pages
- /components - Reusable UI components
- /lib - Utility functions and helpers
- /prisma - Database schema and migrations
\end{lstlisting}
\end{frame}

\begin{frame}[fragile]\frametitle{Initialize \& Understand}
  \begin{itemize}
    \item \textbf{Goal:} Show how OpenCode scans and understands a new project from scratch.
    \item \textbf{Setup:} Open terminal inside any Git repository.
    \item \textbf{Launch:} \lstinline|$ opencode|
    \item \textbf{Command:} \lstinline|> /init|
    \item \textbf{What Happens:} OpenCode scans all local files, infers stack (languages, deps, structure), and writes \texttt{AGENTS.md},  the AI's instruction manual for this project.
    \item \textbf{Key Point:} Every subsequent interaction is grounded in this context,  no hallucinated file paths.
  \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Fix \& Refactor}
  \begin{itemize}
    \item \textbf{Goal:} Demonstrate AI-driven code modification on a real file.
    \item \textbf{Command:} \lstinline|> @file:main.go fix the bug where user input isn't sanitized|
    \item \textbf{Agent Action:} Reads \texttt{main.go}, identifies the unsanitized input vector, proposes a targeted patch.
    \item \textbf{Review Flow:}
      \begin{itemize}
        \item Press \texttt{Tab} → switch to \textbf{Plan} mode: inspect the proposed diff before anything touches disk.
        \item Press \texttt{Tab} → switch to \textbf{Build} mode: apply the change.
      \end{itemize}
    \item \textbf{Key Point:} You stay in control,  the agent proposes, you decide.
  \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Context-Aware Codebase Search}
  \begin{itemize}
    \item \textbf{Goal:} Navigate a large, unfamiliar codebase using natural language.
    \item \textbf{Command:} \lstinline|> @workspace where is the database connection configured?|
    \item \textbf{Agent Action:} Scans the entire workspace, returns exact file(s) and line(s) where DB config lives.
    \item \textbf{Key Point:} Replaces \texttt{grep -r}, \texttt{find}, and guesswork; understands \textit{intent}, not just keywords.
    \item \textbf{Follow-up Capability:} Chain directly into a fix or refactor from the same session.
  \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Add a Feature (Multi-File Editing)}
  \begin{itemize}
    \item \textbf{Goal:} Show coordinated, multi-file changes from a single natural language prompt.
    \item \textbf{Command:} \lstinline|> add a new API endpoint to delete user notes and update the README|
    \item \textbf{Agent Plan:}
      \begin{itemize}
        \item \texttt{routes/notes.go},  add \texttt{DELETE /notes/:id} route
        \item \texttt{handlers/notes.go},  implement handler logic
        \item \texttt{db/notes.go},  add delete query
        \item \texttt{README.md},  document the new endpoint
      \end{itemize}
    \item \textbf{Key Point:} Agent understands cross-file dependencies,  no manual coordination needed.
  \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{Tips for Beginners}
  \begin{itemize}
    \item Be Specific: 
Don't say "Make it better" but say, "Refactor the handleSubmit function to use async/await and add error handling"

    \item Provide Context: Don't say "Fix the bug" but say, "The login form throws 'undefined is not a function' when clicking submit. The error occurs in auth.ts line 45"

    \item Use Plan Mode First: For complex features, ask OpenCode to create a plan before implementation.

    \item Leverage Image Support: Drag and drop screenshots or mockups into the terminal—OpenCode can understand images!

    \item Create Project-Specific Configs: Different projects may need different settings. Use local opencode.json files.
  \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{Why Use OpenCode?}
  \begin{itemize}
    \item \textbf{Privacy-First:} Code never leaves your machine by default; no telemetry, no vendor data retention.
    \item \textbf{Cost Efficiency:} No subscription fees,  pay only for actual API token usage; local models are free.
    \item \textbf{No Vendor Lock-in:} Switch AI providers in seconds via config; one tool, all models.
    \item \textbf{Terminal-Native Speed:} Faster context-switch than IDE plugins; integrates directly into your existing shell workflow.
    \item \textbf{Transparent \& Auditable:} Fully open-source (Apache 2.0),  inspect, fork, or extend anything.
    \item \textbf{Enterprise-Ready:} Supports self-hosted LLMs and air-gapped (physically isolated from all other networks) environments.
    \item \textbf{Bottom Line:} Faster dev loop, zero lock-in, full code privacy,  AI that works \textit{with} your terminal.	
  \end{itemize}
