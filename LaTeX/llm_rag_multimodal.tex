\begin{frame}[fragile]\frametitle{What is Multimodal RAG?}

\textbf{Standard RAG} only retrieves \textit{text}. Real-world documents contain much more:

\begin{itemize}
  \item \textbf{Tables} --- financial reports, lab results, comparison matrices.
  \item \textbf{Images \& Charts} --- diagrams, photos, scanned figures.
  \item \textbf{Structured layouts} --- headings, sections, footnotes, captions.
\end{itemize}

\vspace{0.3em}
\textbf{Multimodal RAG} extends the pipeline to \textit{understand and index all of these modalities}:

\begin{enumerate}
  \item \textbf{Parse} the document preserving its full structure (not just raw text).
  \item \textbf{Embed} text chunks, table cells, and image captions separately.
  \item \textbf{Retrieve} the most relevant modality for each query.
  \item \textbf{Generate} an answer grounded in both text and non-text evidence.
\end{enumerate}

\vspace{0.3em}
{\small \textbf{Docling} (IBM, open-source) is the document parsing backbone we use in this tutorial --- it handles Step 1 with computer vision models.}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Introduction: Traditional PDF Extraction Problems}

\textbf{Why does naive text extraction break RAG?}

      \begin{itemize}
        \item Traditional tools (\texttt{pypdf}, \texttt{PDFMiner}) extract a raw character stream --- they treat a PDF as a flat sequence of bytes, not as a structured document.
        \item \textbf{Tables} become jumbled text: columns merge, row boundaries vanish, numbers lose their column headers.
        \item \textbf{Headers and body} text mix together --- the retriever cannot tell a section title from a paragraph.
        \item \textbf{Images} disappear entirely from the extraction.
        \item \textbf{Result for RAG:} The vector store is polluted with structurally broken chunks. A question about ``Table 3'' retrieves garbled text and the LLM either hallucates or says ``I don't know''.
      \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Docling: IBM's Open-Source Solution}

\textbf{Core idea:} Instead of reading bytes, Docling \textit{sees} the page like a human reader --- using computer vision to recognize layout before extracting text.

      \begin{itemize}
        \item \textbf{Two-model pipeline:} A layout analysis model (DocLayNet) identifies regions (header, body, table, figure); a TableFormer model reconstructs table structure into rows and columns.
        \item \textbf{Structure preservation:} Headers, tables, images, and hierarchies are labelled and kept intact --- making downstream chunks semantically coherent.
        \item Processes documents $\sim$30$\times$ faster than traditional OCR pipelines.
        \item Runs \textbf{locally} on commodity hardware --- no API calls, no data leaving your machine.
        \item Multi-format input: PDF, DOCX, PPTX, XLSX, HTML, images.
        \item Exports to \textbf{Markdown} (LLM-friendly), JSON, and DocTags.
      \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{Docling Architecture: How It Processes a Document}

\textbf{Two-stage computer vision pipeline:}

\begin{enumerate}
  \item \textbf{Layout Analysis Model} (trained on DocLayNet --- 80K+ labelled pages):
  \begin{itemize}
    \item Scans each page as an image.
    \item Classifies every region: \textit{heading, body text, table, figure, caption, footer}.
    \item Outputs a structured bounding-box map of the page.
  \end{itemize}

  \item \textbf{TableFormer Model} (specialized table recognition):
  \begin{itemize}
    \item Receives bounding boxes flagged as ``table''.
    \item Reconstructs row/column structure from pixel patterns.
    \item Exports a proper \texttt{pandas.DataFrame} --- not garbled text.
  \end{itemize}
\end{enumerate}

\vspace{0.3em}
\textbf{Why this matters for RAG:} Structure-aware chunks mean the retriever can find ``Table 3, column Revenue'' rather than a string of jumbled numbers.

\end{frame}

    if "title" in label.lower():
        return 1
    elif "section" in label.lower():
        return 2
    elif "subsection" in label.lower():
        return 3
    else:
        return 4

from typing import List, Dict, Any
import pandas as pd
from docling_core.types.doc import DoclingDocument

class DocumentStructureVisualizer:
    def __init__(self, docling_document: DoclingDocument):
        self.doc = docling_document
	  
def get_document_hierarchy(self) -> List[Dict[str, Any]]:
    hierarchy = []

    if not hasattr(self.doc, "texts") or not self.doc.texts:
        return hierarchy

    for item in self.doc.texts:
        label = getattr(item, "label", None)

        if label and "header" in label.lower():
            text = getattr(item, "text", "")
            prov = getattr(item, "prov", [])
            page_no = prov[0].page_no if prov else None

            hierarchy.append({
                "type": label,
                "text": text,
                "page": page_no,
                "level": self._infer_heading_level(label)
            })

    return hierarchy

\begin{frame}[fragile]\frametitle{Document Chunking Strategy}

\textbf{Why 1000 characters with 100-char overlap?}

\begin{center}
\small
\begin{tabular}{lll}
\hline
\textbf{Chunk size} & \textbf{Retrieval precision} & \textbf{Context per chunk} \\
\hline
$\sim$200 chars & Very high (laser-focused) & Too little; answer may split \\
$\sim$1000 chars & Good balance & $\approx$150--200 tokens; fits context \\
$\sim$3000 chars & Lower & Rich context, but dilutes similarity \\
\hline
\end{tabular}
\end{center}

\vspace{0.3em}
      \begin{itemize}
        \item \textbf{Overlap (100 chars $\approx$ 1 sentence):} Prevents an answer from being cut exactly at a chunk boundary and disappearing from retrieval.
        \item \textbf{Metadata propagation:} Each chunk inherits \texttt{filename} and \texttt{source} from the parent document --- this is how the agent can cite ``Page 4 of report.pdf''.
        \item \textbf{Structure-aware bonus:} Because Docling outputs clean Markdown, the splitter encounters headings (\texttt{\#}) and table rows (\texttt{|}) --- far less noise than raw text.
      \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{LangGraph Agent: ReAct Pattern}
\begin{columns}
    \begin{column}[T]{0.45\linewidth}
      \textbf{What is ReAct?} (Reason + Act)

      Instead of a fixed retrieve-then-generate pipeline, the agent \textit{thinks in a loop}:
      \begin{enumerate}
        \item \textbf{Reason}: ``What do I need to answer this?''
        \item \textbf{Act}: Call \texttt{search\_documents} tool.
        \item \textbf{Observe}: Read the retrieved chunks.
        \item Repeat if still uncertain, or generate final answer.
      \end{enumerate}

      \vspace{0.3em}
      \textbf{Key settings:}
      \begin{itemize}
        \item \texttt{temperature=0} --- deterministic, consistent answers.
        \item \texttt{MemorySaver} --- stores conversation turns so follow-up questions work.
      \end{itemize}
    \end{column}
    \begin{column}[T]{0.55\linewidth}
      \begin{lstlisting}[language=python, basicstyle=\tiny]
SYSTEM_PROMPT = """You are a helpful document intelligence assistant. You have access to documents that have been uploaded and processed.

GUIDELINES:
- Use the search_documents tool to find relevant information
- Keep it simple: one well-crafted search is usually sufficient
- Only search again if the first results are clearly incomplete
- Provide clear, accurate answers based on the document contents
- Always cite your sources with filenames
- If information isn't found, say so clearly
- Be concise but thorough
...
"""
def create_documentation_agent(tools, model_name="gpt-4o-mini"):
    llm = ChatOpenAI(model=model_name, temperature=0)
    memory = MemorySaver()
    agent = create_react_agent(
        llm, tools=tools,
        prompt=SYSTEM_PROMPT,
        checkpointer=memory
    )
    return agent
      \end{lstlisting}
    \end{column}
  \end{columns}
\end{frame}

\begin{frame}[fragile]\frametitle{Streaming Response Implementation}

\textbf{Why stream?} A ReAct agent may take 3--10 seconds before producing output (retrieval + reasoning). Without streaming, the UI shows a blank screen and users assume the app is broken.

      \begin{itemize}
        \item \textbf{Status transitions} give live feedback: \texttt{"Thinking..."} $\rightarrow$ \texttt{"Searching documents..."} $\rightarrow$ \texttt{"Generating answer..."}
        \item \textbf{\texttt{stream\_mode="messages"}} streams at token level --- you process the LLM output word-by-word rather than waiting for the complete response.
        \item \textbf{Tool-call detection:} The stream emits both tool-call events (when the agent searches) and text events (when it generates). We inspect each chunk's type to update the status indicator correctly.
        \item \textbf{\texttt{st.write\_stream()}} handles the token accumulation and renders the typing effect in Streamlit automatically.
        \item The complete response is appended to \texttt{conversation\_history} after streaming finishes, so follow-up questions have full context.
      \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{Extension Opportunities}
      \begin{itemize}
        \item \textbf{Batch processing:} Ingest entire document folders into a persistent ChromaDB collection.
        \item \textbf{Persistent vector store:} Save and reload the ChromaDB index across sessions (no re-embedding on restart).
        \item \textbf{Alternative OCR engines:} Swap in EasyOCR or Tesseract for non-Latin scripts.
        \item \textbf{Vision-Language Models:} Use GPT-4o or LLaVA to answer questions directly about chart images (not just captions).
        \item \textbf{Graph RAG:} Build a knowledge graph from extracted entities in the document hierarchy for multi-hop Q\&A.
        \item \textbf{Deployment:} Containerize with Docker; deploy to Streamlit Cloud or a private server.
        \item \textbf{Document comparison:} Embed multiple versions of a document; query for differences.
      \end{itemize}
