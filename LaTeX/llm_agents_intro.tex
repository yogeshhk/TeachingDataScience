    \item Examples of what users now expect:
    \begin{itemize}
        \item Instead of listing leads $\rightarrow$ \textbf{email them directly}
        \item Instead of summarizing docs $\rightarrow$ \textbf{file and create workflow tasks}

    \end{itemize}
    \item This shift from \textbf{information} to \textbf{action} defines the agent era
\end{itemize}

\begin{frame}[fragile]\frametitle{The ReAct Loop: Reason \textrightarrow{} Act \textrightarrow{} Observe}
      \begin{itemize}
        \item An agent is \textbf{not} a one-shot responder: it runs a continuous loop
        \item Think of it as a \texttt{while} loop that keeps iterating until the task is done:
        \begin{itemize}
            \item \textbf{Thought}: analyse the situation and plan the next step
            \item \textbf{Action}: call a tool to execute the plan
                  (search, booking API, email, code runner \ldots)
            \item \textbf{Observation}: examine what the tool returned
        \end{itemize}
        \item The cycle repeats: Thought $\rightarrow$ Action $\rightarrow$ Observation
        \item The loop \textbf{adapts}: if an action fails, the next Thought incorporates that failure
        \item This is the \textbf{ReAct} paradigm (Yao et al., 2022): \textbf{Re}asoning + \textbf{Act}ing
        \item Uses \textbf{memory} to keep conversation history and past actions
        \item May involve a \textbf{human-in-the-loop} for safety in high-stakes steps
      \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{ReAct in Practice: Thought + Action}
      \begin{itemize}
        \item Before \emph{every} action an agent verbalises a \textbf{thought trace}:
            \begin{itemize}
                \item \emph{``User wants the weather in New York. I have a weather tool.''}
                \item \emph{``My next move: call \texttt{get\_weather("New York")}.''}
            \end{itemize}
        \item This deliberation makes agents \textbf{more than reactive scripts}
        \item The magic lies in \textbf{tools} and \textbf{function calling}:
            \begin{itemize}
                \item \texttt{send\_email(to="alice@co.com", body="...")}
                \item \texttt{fetch\_crm\_records(query="open leads")}
                \item \texttt{schedule\_meeting(time="Tuesday 14:00")}
            \end{itemize}
        \item \textbf{Mental model:} LLM = brain \quad Tools = hands
        \item Without tools, agents just talk.\ With tools, they \textbf{act}.
      \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{Two Ways Agents Use Tools}
      \begin{itemize}
        \item \textbf{JSON Agent}: Outputs a structured work-order (JSON) for another system to execute
        \begin{itemize}
            \item Requires an external system to parse and run the JSON
            \item More portable, but adds a layer of indirection
        \end{itemize}
        \item \textbf{Code Agent}: Directly writes and runs Python (or other) code
        \begin{itemize}
            \item More direct and powerful; code can do loops, conditionals
            \item Taps into thousands of existing libraries and APIs
            \item Easier to debug; results are grounded in actual execution
        \end{itemize}
        \item Both patterns are valid; choice depends on safety and capability needs
      \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{The Context Window: Why It Limits Agents}
      \begin{itemize}
        \item Every LLM call accepts a fixed maximum number of \textbf{tokens} (word pieces)
        \item A token $\approx$ 0.75 English words; GPT-4o supports $\sim$128k tokens; Claude 3.5 Sonnet $\sim$200k
        \item \textbf{Why this matters for agents:}
        \begin{itemize}
            \item Each ReAct loop iteration appends Thought + Action + Observation to the context
            \item A 10-step agent task may consume \textbf{10$\times$} the tokens of a single LLM call
            \item Longer contexts = higher \textbf{cost} and higher \textbf{latency} per call
        \end{itemize}
        \item \textbf{Intuition:} The context window is like a whiteboard: it holds only what fits right now. When the board is full, the agent must decide what to \emph{erase} (summarise into episodic memory) and what to \emph{keep}
        \item This is why \textbf{memory management} and \textbf{summarisation} are critical agent skills
      \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Memory: Four Types Agents Use}
      \begin{itemize}
        \item Memory lets an agent ``remember'': without it, every turn starts blank
        \item Four types are commonly used in practice:
        \begin{itemize}
            \item \textbf{In-Context (Sensory)}: the current conversation window; fast but limited in size
            \item \textbf{Episodic}: summaries of past sessions stored in a database; survives restarts
            \item \textbf{Semantic}: facts about the world or the user (e.g.\ ``user prefers Python''); stored in a vector DB
            \item \textbf{Procedural}: the agent's instructions / system prompt; rarely changes
        \end{itemize}
        \item \textbf{Intuition:} Think of in-context memory as your desk, episodic as your diary, semantic as an encyclopaedia, and procedural as your job description
        \item Most frameworks (Agno, LangChain, CrewAI) implement all four
      \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{Agentic RAG}
      \begin{itemize}
        \item \textbf{Traditional RAG}: Fetch relevant documents once, then answer
        \item \textbf{Agentic RAG} turns retrieval itself into an agent-driven task
        \item Think of it as a \textbf{master researcher}: it doesn't just do one search
        \item Instead, it:
        \begin{itemize}
            \item Rewrites queries for better recall
            \item Runs multiple targeted searches
            \item Validates accuracy before using results
            \item Pulls from both private data and the public web
        \end{itemize}
        \item Significantly improves answer quality for complex, multi-faceted questions
      \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{Multi-Agent Systems: Digital Teams}
      \begin{itemize}
        \item Complex problems require specialised skill sets: one agent is not enough
        \item Solution: Build \textbf{teams} of specialised agents
        \item A \textbf{Manager agent} acts as project lead and breaks down big tasks
        \item Delegates sub-tasks to \textbf{specialist agents} with focused skills
        \begin{itemize}
            \item Web agent handles online searching
            \item Code agent handles computation
            \item Writer agent drafts final reports
        \end{itemize}
        \item Manager focuses on the big picture and final integration
        \item Digital division of labour for complex, multi-step problem solving
      \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{Agentic Workflow Design Patterns}

\begin{frame}[fragile]\frametitle{Mitigating Predictability Challenges}
    \begin{itemize}
        \item Choose \textbf{reasoning models} when possible (e.g.\ Gemini 2.5 Flash, o1)
        \item Ground the LLM in \textbf{trusted data} to reduce hallucinations (RAG, search)
        \item Guide the agent to \textbf{think step-by-step} (chain-of-thought, few-shot prompting)
        \item Use \textbf{state management} and memory to track progress
        \item Implement checks in the workflow: tool-call validation, safety filters, logic guards
        \item Continuously \textbf{test and evaluate} your agent on representative scenarios
    \end{itemize}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Mitigating Stability Challenges}
    \begin{itemize}
        \item Use consistent tool abstractions via \textbf{Model Context Protocol (MCP)}
        \item Write \textbf{detailed system prompts}; include precise tool descriptions
        \item Implement \textbf{error-handling}: circuit-breakers, retries, human hand-off
        \item Utilise \textbf{state management} (especially workflow status) to recover from interruptions and prevent infinite loops
        \item Be willing to refactor as framework features and tool interfaces evolve
    \end{itemize}

\begin{frame}[fragile]\frametitle{Mitigating Operations Challenges}
    \begin{itemize}

        \item Build robust \textbf{test and evaluation datasets}; automate tests on every commit
        \item Gather key metrics: token throughput, response code distribution; create alerts
        \item Leverage framework \textbf{tracing features} to find and address latency bottlenecks

\begin{frame}[fragile]\frametitle{Model Context Protocol (MCP): What and Why}
      \begin{itemize}
        \item \textbf{MCP} is an open standard (Anthropic, 2024) that defines a \emph{common language} between an LLM and its tools
        \item \textbf{Before MCP}: every framework invented its own way to describe tools to the LLM: fragmented and non-portable
        \item \textbf{With MCP}: a tool exposes a machine-readable \emph{schema} (name, description, parameters); any MCP-aware agent can call it without custom glue code
        \item \textbf{Intuition:} MCP is to AI tools what USB is to hardware peripherals: one standard connector, any device
        \item Benefits:
        \begin{itemize}
            \item Plug any MCP tool into any MCP-aware framework (Agno, LangChain, Claude\ldots)
            \item Reduces integration effort from days to minutes
            \item Enables auditable, versioned tool interfaces
        \end{itemize}
        \item Already supported by Agno, LangGraph, OpenAI Agents SDK, Google ADK, and Cursor
      \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Importance of Evaluation in Agentic Workflows}
      \begin{itemize}
        \item Across many teams building agentic systems, a clear pattern emerges:
        \item The biggest predictor of success is \textbf{disciplined evaluation (evals)}
        \item Strong eval practices lead to more effective, reliable agents
        \item Weak evals result in inefficient development cycles
        \item Ability to design and run evals is a \textbf{core skill} for agent builders
      \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Why Evals Are Hard to Design Upfront}
      \begin{itemize}
        \item Agentic workflows have many \textbf{unforeseen} failure modes
        \item Difficult to anticipate issues before real deployment
        \item Example: a customer order inquiry agent might unexpectedly mention competitors
        \item Problems often surface only after observing real outputs
        \item Best practice: \textbf{build first, evaluate later}
        \item Iterative inspection drives meaningful improvement
      \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Subjective Evals with LLMs as Judges}
      \begin{itemize}
        \item Many evaluation criteria are inherently \textbf{subjective}
        \item Free-text outputs resist simple rule-based scoring
        \item Use an \textbf{LLM as a judge} for quality assessment
        \item Example: prompt a judge LLM to rate essay quality from 1 to 5
        \item Apply the judge across many generated outputs to track trends
        \item Scores can indicate whether iterations are improving the system
      \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Limitations of LLM-Based Scoring}
      \begin{itemize}
        \item LLMs are \textbf{weak at fine-grained numeric ratings}
        \item 1--5 scale scores are often inconsistent and unreliable
        \item Technique is useful as an \textbf{initial rough baseline} only
        \item Better evaluation methods exist (e.g.\ pairwise comparison)
        \item Still useful for quick, directional comparisons early in development
      \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Types of Agentic AI Evaluations}
      \begin{itemize}
        \item \textbf{Objective evals}: code-based checks (did the agent mention a competitor?)
        \item \textbf{Subjective evals}: LLM judges for quality, tone, helpfulness
        \item \textbf{End-to-end evals}: measure full agent output quality for a complete task
        \item \textbf{Component-level evals}: assess individual steps (retrieval, tool call, generation)
        \item Using different eval types helps \textbf{isolate and fix} specific weaknesses
      \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Error Analysis and Traces}
      \begin{itemize}
        \item Inspect \textbf{intermediate agent outputs} (traces) step by step
        \item Review each reasoning step and tool call in the workflow
        \item Identify exactly \textbf{where} behaviour deviates from expectations
        \item Manual reading of traces reveals improvement opportunities that metrics miss
        \item Known as \textbf{error analysis}: an essential skill for agent builders
        \item Strongly linked to effective eval design and iteration
      \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{The ReAct Paper: Why It Matters}
\begin{itemize}
    \item \textbf{ReAct} (Yao et al., 2022) formalised the \textbf{Reason + Act} loop
    \item Key insight: interleaving \emph{thought traces} with \emph{tool actions} outperforms either alone
    \item Standard prompting: only generates text
    \item Chain-of-Thought (CoT): reasons but does not act
    \item ReAct: reasons \textbf{and} acts, then observes results and reasons again
    \item On HotpotQA multi-hop QA, ReAct raised task success by $\sim$34 percentage points vs.\ CoT alone; gains vary by task type
    \item \textbf{Intuition:} thinking aloud before each action helps the agent \textbf{stay on track} across many steps: just as a good engineer comments their code
\end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{Python AI Agent Frameworks Overview}
    \begin{itemize}
        \item Python offers several frameworks for building single and multi-agent AI systems
        \item These frameworks support workflows, tool use, reasoning, and human-in-the-loop
        \item Common applications: autonomous assistants, research co-pilots, workflow orchestration
        \item Frameworks differ in level of abstraction: code-level, low-code, and no-code options
    \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{LangChain / LangGraph}
    \begin{itemize}
        \item \textbf{Type:} Multi-agent, graph-based framework
        \item \textbf{Built on:} LangGraph for agent state management and orchestration
        \item \textbf{Features:}
              \begin{itemize}
                  \item Agent-centric architecture with workflow graphs
                  \item Human-in-the-loop integration via LangSmith observability
                  \item Supports OpenAI, Anthropic, Gemini, and local models
              \end{itemize}
    \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Pydantic-AI}
    \begin{itemize}
        \item \textbf{Type:} Lightweight single/multi-agent framework
        \item \textbf{Focus:} Strong type safety using Pydantic for structured inputs/outputs
        \item \textbf{Features:}
              \begin{itemize}
                  \item Declarative agent definition with automatic validation
                  \item Easy LLM orchestration and chaining
                  \item Built for developers needing data integrity guarantees
              \end{itemize}
        \item \textbf{Ideal for:} AI applications requiring schema consistency and typed reasoning
    \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Crew AI}
    \begin{itemize}
        \item \textbf{Type:} Multi-agent coordination framework
        \item \textbf{Concept:} A ``crew'' of specialised agents working collaboratively
        \item \textbf{Features:}
              \begin{itemize}
                  \item Role-based agent interactions with shared memory
                  \item Supports autonomous workflow creation
                  \item Integrates with LangChain, OpenAI, and custom tools
              \end{itemize}
        \item \textbf{Use Case:} Complex multi-role tasks (research, planning, content generation)
    \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Autogen}
    \begin{itemize}
        \item \textbf{Developed by:} Microsoft Research
        \item \textbf{Type:} Multi-agent conversational framework
        \item \textbf{Features:}
              \begin{itemize}
                  \item Agent-to-agent and human-in-loop communication
                  \item Supports function calling and code execution
                  \item Built-in memory and orchestration logic
              \end{itemize}
        \item \textbf{Example:}
    \end{itemize}
\begin{lstlisting}[language=Python]
from autogen import AssistantAgent, UserProxyAgent

assistant = AssistantAgent("assistant", llm="gpt-4")
user = UserProxyAgent("user", code_execution=True)

user.initiate_chat(assistant, message="Build a weather app.")
\end{lstlisting}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{OpenAI Agents SDK}
    \begin{itemize}
        \item \textbf{Type:} Hosted agent platform by OpenAI (2024+)
        \item \textbf{Features:}
              \begin{itemize}
                  \item Create, configure, and deploy GPT-based agents via API
                  \item Integrates with files, APIs, tools, and memory natively
                  \item Supports human feedback and evaluation workflows
              \end{itemize}
        \item \textbf{Example Use:} Deploy a custom assistant with a knowledge base and tools
    \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Google ADK (Agent Development Kit)}
    \begin{itemize}
        \item \textbf{Developed by:} Google DeepMind / Gemini ecosystem
        \item \textbf{Type:} Multi-agent and workflow orchestration framework
        \item \textbf{Features:}
              \begin{itemize}
                  \item Integrates Gemini models with tool-use capabilities
                  \item Supports perception, memory, planning, and dialogue
                  \item Human-in-the-loop and fully autonomous modes
                  \item Built for scalable, production-grade AI agents

        \item \textbf{Use Case:} Gemini-powered assistants and multi-agent reasoning systems
    \end{itemize}
\end{frame}

\begin{frame}[fragile]\frametitle{Comparison Summary}
    \begin{itemize}
        \item \textbf{LangChain/LangGraph:} Graph-based, modular, integrates with LangSmith for tracing
        \item \textbf{Pydantic-AI:} Strong typing, data-safe, schema-consistent orchestration
        \item \textbf{CrewAI:} Role-based multi-agent collaboration, high-level abstraction
        \item \textbf{Autogen:} Conversational multi-agent system by Microsoft Research
        \item \textbf{OpenAI Agents SDK:} Hosted, GPT-native, low-code deployment
        \item \textbf{Google ADK:} Gemini-native, multi-agent orchestration suite
        \item \textbf{Agno:} Lightweight, 23+ model providers, production-ready with AgentOS
    \end{itemize}
\end{frame}
